# Phase 5A - Threshold-Based Detection Rule

For this part of phase 5, I created detection rules that were based on thresholds.

My focus is on detecting abnormal behavior in SMB activity, particularly in activity rate and volume. I particularly use the negotiation command event for this purpose. The reason for using the negotiation command is that it's an entry point to the SMB protocol, and I can use it to drive these detection hypotheses.

1. Sustained elevated SMB negotiate volume within a 12-hour period may indicate increased scanning, an increase in campaigns, or an extended session attempt. (Volume-based)

2. Elevated SMB command activity within a 5-minute interval may indicate brute-force authentication attempts, rapid enumeration, automated probing, or other abnormal SMB interactions. (Rate-based)

Detection rules do not directly confirm these hypotheses. Instead, they provide investigative starting points that may indicate activities consistent with the hypotheses.

## Detection Development Framework

For this phase, I followed the following model to create rules. What I found useful about this model is that it helps me not only to define the purpose of the rule, but also to scope the data set and granularity, which I found very important for visualization and threshold determination.  
### Rule Creation Model Template
#### 1. Define Behavior
- Excessive SMB command activity (daily volume)
#### 2. Define Detection Metric
- SMB command count per hour
#### 3. Define Baseline and Behavioral Segmentation
- 14-day historical window
- derive normal range or percentile (P95/P99)
#### Behavioral-Based Segmentation:
- a - (number)
- b - (number)
- c - (number)
#### 4. Define Rule Execution
- evaluation interval: 1 hour
- look-back: small buffer (e.g. 3 min)
#### 5. Create Rule
- threshold-based alert on aggregated metric
#### 6. Validate and Tune
- check against known spikes
- adjust threshold or window if needed

## Rule 1: SMB Negotiate Command - 12h Volume Threshold Exceeded

#### Hypothesis:
Sustained elevated SMB negotiate volume in a 12-hour period may indicate increased scanning, an increase in campaigns, or an extended session attempt. (Volume-based)

## Rule Creation Model

### 1. Define Behavior
- Excessive SMB Negotiate activity in a 12-hour period.
### 2. Define Detection Metric
- SMB Negotiate command count per 12-hour period.
```
smb.command : *NEGOTIATE*
```
### 3. Define Baseline and Behavioral Segmentation
- 30-day historical window (May 1, 2026 to May 31, 2026)
- derive normal range

#### Behavioral-Based Segmentation:
- Idle/one-off Activity: 0 ~ 100            
    - 0 ~ 8 session / hour
- Minor Activity: 100 ~ 1500                
    - 8 ~ 125 session / hour

- Elevated Activity: 5000 ~ 8000

Based on the plotted graph of the sorted data points, the slope changes drastically around 5000. Considering this graph and the fact that this environment is a honeypot where constant activity, such as probing or scanning, can be considered normal. And I derived 2 rules based on the finding:
1. Elevated Activity with 5000 as the threshold.

Based on the overall alerts, my observation is that the rules captured the target activity level as expected, including the activity that I first chose for this case study.

Since it's a 12-hour aggregation, it's triggered at the end of the 12-hour bucket data evaluation. So, it may be perceived as alerts arriving visually post-activity, and one may think it's not desired behavior. Therefore, it is important to understand how the alert system operates across different time intervals. The rule evaluates the preceding 12-hour window during each execution cycle. As a result, alerts are generated after the aggregation window has completed, rather than at the exact moment activity begins.

At this stage, I do not plan to further tune the threshold, as the generated alerts aligned with the intended activity categories during preview validation. Future tuning may be driven by the outcomes of investigations, additional observation periods, or changes in the operational environment.

## Rule 2:  SMB Negotiate Command - 5-Minute Activity Rate Threshold Exceeded

#### Hypothesis:
Elevated SMB command activity within a 5-minute interval may indicate brute-force authentication attempts, rapid enumeration, automated probing, or other abnormal SMB interactions.
## Rule Creation Model

### 1. Define Behavior
- Excessive SMB Negotiate command rate (intensity)
### 2. Define Detection Metric
- SMB Negotiate command count 5 min
```
smb.command : *NEGOTIATE*
```
### 3. Define Baseline and Behavioral Segmentation
- 7-day historical window (May 3 - May 10)
- derive normal range
#### Behavioral-Based Segmentation:
- Idle/Very Low Rate: 0 ~ 100              
    - 0 ~ 4 sessions per min
- Low Rate: 100 ~ 200
    - 4 ~ 8 sessions per min
- Mid Rate: 200 ~ 400
    - 8 ~ 16 sessions per min
- High Rate: 400 ~ 700
    - 16 ~ 32 sessions per min
- Very High Rate: > 700
    - \> 32 sessions per min

For this rule, 500 was chosen as the threshold because the data sample shows a drastic increase in rate after 500, which can be treated as an anomaly. Also, given that this environment is a honeypot, it's common to see higher activity rates.

- Look-back: no buffer, as even a small duration could affect the rate (e.g., 0 min)

### 5. Create rule
- Threshold-based alert on aggregated SMB Negotiate command

## Detection Rule Recommendation
Based on these rules, for threshold-based rules, my recommendation for basic threshold rules is as follows:
1. Activity Coverage: **Volume and Rate**
2. Protocol Behavior: **SMB Negotiation Command**
3. Behavioral Segmentation: **Based on activity behavior**
4. Environmental Activity Factor: **Honeypot**

### Activity Coverage
It's recommended to create at least two types of rules that would capture activities by volume and rate. Since volume-based detection rules tend to have longer time intervals, alerts may be triggered after activity; however, the rate-based detection rule may complement such activity and be triggered sooner if the activity has both characteristics.

During validation, I observed that the dominant source IP identified in previous phases generated sufficient SMB negotiate activity to trigger the 12-hour volume rule, but it nearly fell below the threshold of the 5-minute activity rate rule. As a result, the volume-based and rate-based rules provide complementary coverage, improving the likelihood of detecting different operational patterns within the honeypot environment. The dominant IP narrowly fell below the 530 threshold used in the final version of the rate-based rule, highlighting the trade-off between reducing alert volume and maintaining detection coverage. I would not conclude that it was a bad decision, but it can be a factor in threshold tuning for future reference and in validating the rule against properties of interest, such as IPs, ASs, or countries.

### Protocol Behavior
My original thought was to capture all SMB commands, but I realized that I was looking at the graph based on the negotiate command. Instead of reworking the threshold creation, I decided that a negotiate command suits my purpose, since it's the entry command executed in the SMB session. So it indicates SMB session activity. Detecting all SMB commands would provide overall activity, but since the number of commands varies by session, it could be tricky to standardize or create segmentations without context from SMB command progression or flow. Therefore, my recommendation is to use the negotiate command.

### Behavioral Segmentation
It would be helpful to create segments based on what the behavior would look like, rather than just dividing the number. I started to work on simply dividing the activities into "low", "medium", and "high". However, making segmentation at a more detailed behavioral level helped me decide what to include in the baseline and what not to.

### Environmental Activity Factor
This server, being the honeypot, was considered. In a honeypot environment, it's common to observe scanning, probing, and reconnaissance behavior using automated tools. So, a higher activity rate would be normal for this environment.

## Limitation
- Those rules created are tested only in the Rule preview feature. My working environment is that event data or indices are restored from past events. So they are not ingested in real time, and the rule system would not run on such data. There is a possibility that I could re-ingest events as new to test rules, but I have not researched how to do it.
- I am not able to run those on the production honeypot since rules require enabling the Kibana security feature. Doing so requires additional setup, which may break my production environment.

## Future Improvement
- Convert / Improve the threshold-based rule to use a percentile
- Consider other SMB commands and behavior for the threshold-based rule.
- Validate the rules against live telemetry to evaluate alert fidelity and operational usefulness.
